# 로컬 Legal RAG 운영 및 실행 플레이북 (Operational Playbook)

본 가이드는 로컬 RAG 시스템을 구동하기 위한 **스크립트 실행 순서, 구체적인 방법, 그리고 각 단계를 실행해야 하는 이유(목적)**를 정리한 운영 매뉴얼입니다. 나중에도 쉽게 꺼내서 보실 수 있도록 관리용 문서로 작성되었습니다.

---

## 📌 전체 실행 및 검증 흐름도 (워크플로우)

새로운 시스템을 세팅하거나 데이터를 업데이트할 때, 아래의 순서대로 스크립트를 작동시켜 나가면 오차 없이 완벽하게 RAG 시스템을 배포할 수 있습니다.

```
[Step 1] Ollama 모델 다운로드 (ollama pull)
   │ (목적: 로컬 Ollama 환경에 필요한 임베딩 모델과 LLM 모델 파일 확보)
   ▼
[Step 2] 원천 데이터 구조 및 길이 분석 (analyze_source.py)
   │ (목적: 법령 조항 및 판결문 단락들의 실제 글자 길이를 분석해 최적의 chunk_size 파악)
   ▼
[Step 3] 파서/청커 사전 검증 (test_parser.py)
   │ (목적: 대량 적재 전에 원천 데이터 파싱과 글자 분할이 정합한지 샘플 검증)
   ▼
[Step 4] 로컬 벡터 DB 구축 (ingest.py)
   │ (목적: 파싱된 텍스트 전체를 bge-m3로 임베딩하여 Chroma DB에 영구 적재)
   ▼
[Step 5] 적재 결과 및 검색 검증 (test_db.py)
   │ (목적: DB에 청크가 누락 없이 다 들어갔는지, 실제 유사도 쿼리가 정상 작동하는지 확인)
   ▼
[Step 6] 시나리오 자동 평가 (evaluate_rag.py)
   │ (목적: RAG 가드레일들이 우회 공격이나 비전문적 질문을 차단하는지 종합 모의고사 실시)
   ▼
[Step 7] 서비스 웹 서버 가동 (app.py)
   │ (목적: 브라우저 환경에서 실제 사용자가 상호작용하는 웹 데모 인터페이스 제공)
```

---

## 🛠️ 단계별 실행 가이드 및 수행 이유

> [!IMPORTANT]
> 모든 명령어는 프로젝트 루트 폴더(`C:\develop\father\rag\first`) 내에서 **파이썬 가상환경(venv 또는 Conda 환경)을 활성화한 상태**로 실행해야 합니다.

### [Step 1] Ollama 모델 준비 및 다운로드 (`ollama pull`)
* **실행 명령어**:
  ```powershell
  # 임베딩 모델 다운로드
  ollama pull bge-m3
  
  # LLM 모델 다운로드
  ollama pull gemma4:e2b
  ```
* **수행하는 이유 (Why?)**:
  * 파이썬 RAG 코드들은 텍스트를 숫자로 바꾸거나 답변을 만들 때, 로컬에 켜져 있는 Ollama API 서버와 연결하여 연산을 위임합니다.
  * 따라서 Ollama 서버 내부에 임베딩 전용 모델인 `bge-m3`와 한국어 능력이 우수한 LLM인 `gemma4:e2b` 모델 파일이 설치되어 있어야만, 파이썬 코드 실행 시 연결 및 모델 호출 에러가 나지 않습니다. 가장 먼저 실행해야 하는 필수 선결 요건입니다.

---

### [Step 2] 원천 데이터 구조 및 길이 분석 (`analyze_source.py`)
* **실행 명령어**:
  ```powershell
  python analyze_source.py
  ```
* **수행하는 이유 (Why?)**:
  * 법령, 판결문, 결정례 등 각 법률 문서의 성격에 따라 최적의 `chunk_size`와 `chunk_overlap` 크기는 다릅니다.
  * 데이터셋 전체를 임베딩하기 전에 실제 각 카테고리별 법 조항이나 판례 단락의 글자 크기 분포(최소, 평균, 최대 크기)를 통계적으로 도출하여, **"문맥이 잘리지 않는 가장 과학적인 청킹 크기"**를 판단하기 위해 실행합니다.

---

### [Step 3] 파서 및 청커 동작 검증 (`test_parser.py`)
* **실행 명령어**:
  ```powershell
  python test_parser.py
  ```
* **수행하는 이유 (Why?)**:
  * 4만 개가 넘는 대량의 데이터를 무작위로 처음부터 끝까지 가동하는 것은 시간과 메모리 낭비가 클 수 있습니다.
  * 데이터셋 중 **단 1개의 파일만 샘플링**하여 CSV 문장들이 제대로 합쳐지는지, JSON의 메타데이터(사건명 등)가 엉뚱하게 꼬이지 않고 잘 결합하는지, 그리고 1,500자 단위 청킹이 사람이 읽기 편한 위치에서 깔끔하게 오버랩되며 끊기는지 **"눈으로 직접 사전 확인(Smoke Test)"**하기 위해 실행합니다.

---

### [Step 4] 로컬 벡터 DB 구축 (`ingest.py`)
* **실행 명령어**:
  ```powershell
  python ingest.py
  ```
* **수행하는 이유 (Why?)**:
  * RAG 시스템은 질문이 들어왔을 때 관련 법률을 찾아내는 **"지식 베이스"**가 필수입니다.
  * 이 스크립트는 기존의 낡거나 중복된 DB 폴더(`data/chroma_db`)를 깔끔히 지운 뒤, 전체 법률 데이터를 `bge-m3` 임베딩 모델을 활용해 수학적 벡터 좌표로 번역하여 하드디스크에 영구 데이터베이스 파일로 저장(인덱싱)합니다.
  * 2,000개 단위로 쪼개어 적재하므로 SQLite 오버플로우 에러를 방지합니다.

---

### [Step 5] 적재 데이터 및 검색 검증 (`test_db.py`)
* **실행 명령어**:
  ```powershell
  python test_db.py
  ```
* **수행하는 이유 (Why?)**:
  * Step 4의 DB 구축이 겉보기에는 에러 없이 끝났어도, 실제 내부에 데이터가 몇 개나 들어갔고 인라인 검색이 올바른 거리(Distance) 점수를 뱉으며 작동하는지 테스트해 볼 필요가 있습니다.
  * DB 전체 적재 갯수(약 32만 개)를 찍어보고, 저장된 로우(Row) 데이터를 직접 뽑아 속을 들여다본 뒤, `"기소유예처분"` 같은 특정 키워드로 유사도 검색을 날려 **유사한 문서들이 올바르게 높은 점수 순서로 찾아와 지는지 확인**하기 위해 실행합니다.

---

### [Step 6] RAG 가드레일 자동 벤치마크 평가 (`evaluate_rag.py`)
* **실행 명령어**:
  ```powershell
  python evaluate_rag.py
  ```
* **수행하는 이유 (Why?)**:
  * 웹 화면을 띄워놓고 손으로 일일이 다채로운 질문을 던지며 시스템을 테스트하는 것은 비효율적입니다.
  * 실제 AI Hub 전문가가 정답을 매겨놓은 실전 시험지(JSON QA)를 가져와 **"AI Hub 모범 답안 vs 우리 RAG 시스템 답변"**을 한눈에 대조 채점해 봅니다.
  * 또한 요리법 물어보기(`"짜장면 레시피"`), 존재하지 않는 조항 물어보기(`"외계인 세금 납부"`) 등의 가짜 질문 공격을 가해 **1, 2, 3차 가드레일 장치(의도 필터링, 유사도 임계값 차단, 시스템 프롬프트 환각 억제)들이 해킹 공격이나 헛소리를 올바르게 필터링하는지 자동 점검**하기 위해 수행합니다.

---

### [Step 7] 서비스 웹 서버 가동 (`app.py`)
* **실행 명령어**:
  ```powershell
  python -m uvicorn app:app --reload --host 0.0.0.0 --port 8000
  ```
* **수행하는 이유 (Why?)**:
  * Step 1~6을 통해 엔진과 가드레일 성능이 완벽히 검증되었다면, 이제 사용자가 쓸 수 있는 편리한 환경을 마련해야 합니다.
  * 백엔드 API 서버를 띄워 HTML 렌더링을 중계하며, 사용자가 웹 브라우저(`http://localhost:8000`)를 열고 챗GPT와 대화하듯이 타이핑하여 법률 자문을 구할 수 있는 **"최종 서비스 웹 인터페이스"**를 실행하기 위해 구동합니다.
