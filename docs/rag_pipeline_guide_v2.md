# 로컬 Legal RAG 시스템 파이프라인 워크플로우 가이드 (v2)

본 문서는 AI Hub 법률 데이터셋을 가공하여 **하이브리드 세만틱 로컬 벡터 데이터베이스(v2)**를 구축하고, 삼중 가드레일을 통해 질문에 답변하는 RAG(Retrieval-Augmented Generation) 시스템의 전체 수명 주기(Lifecycle)와 단계별 순서를 설명합니다.

---

## 📌 RAG v2 전체 아키텍처 흐름도

```mermaid
flowchart TD
    %% Phase 1: Ingestion
    subgraph Ingestion_Phase [1단계: 지식 베이스 구축 (ingest_v2.py)]
        A[원천 데이터 CSV + JSON 메타데이터] --> B[1. 파싱 Parser: 본문 결합 및 메타데이터 매핑]
        B --> C[2. 청킹 Splitter V2: 정규식 1차 분리 및 임베딩 유사도 2차 분할]
        C --> D[3. 임베딩 Embedding: Ollama bge-m3 호출]
        D --> E[4. 벡터 적재 Vector Store: Chroma DB v2에 2,000개 배치 단위 저장]
    end

    %% Phase 2: Inference
    subgraph Inference_Phase [2단계: 질의응답 및 추론 (app_v2.py)]
        F[사용자 질문 입력] --> G{1차 가드레일: 법률 의도 판단?}
        G -- No --> H[즉시 차단 및 거절 메세지 반환]
        G -- Yes --> I[bge-m3로 질문 임베딩 변환]
        I --> J[Chroma DB v2 유사도 검색 Top 10]
        J --> K{3차 가드레일: 검색 스코어 <= 1.1?}
        K -- No --> L[참고 문서 배제 / 빈 근거 전달]
        K -- Yes --> M[가드레일 통과한 근거 문서 확보]
        M --> N[2차 가드레일 & LLM: gemma4:e2b 호출]
        L --> N
        N --> O[출처 인라인 표기 및 모법 답변 생성]
    end

    E -.-> J
```

## 🗂️ 1단계: 지식 베이스 구축 (Ingestion Lifecycle v2)
`python ingest_v2.py`를 실행할 때 백엔드에서 수행되는 데이터 변환 순서입니다.

### [Step 1] 파싱 및 메타데이터 결합 (Parsing)
* **역할**: 흩어져 있는 원천 법률 텍스트와 사건 정보를 하나로 묶어 정형화합니다.
* **상세**: 
  1. `01.원천데이터` 폴더의 CSV 파일들을 한 문장씩 읽어 전체 판례 본문(`full_text`)으로 병합합니다.
  2. 사전에 색인해 둔 `02.라벨링데이터` 폴더의 JSON 파일들과 1:1 매칭하여 **사건번호, 판례명, 선고일자, 문서 유형**을 가져옵니다.

### [Step 2] 청킹 / 하이브리드 세만틱 분할 (Chunking v2)
* **역할**: 대용량 문장 덩어리를 의미 훼손과 단락 잘림 없이 완벽한 컨텍스트 조각으로 분리합니다.
* **상세**:
  1. **1차 분할 (정규식 전방 탐색)**: 
     글자가 삭제되는 일 없이 원본을 100% 보존하기 위해 정규식 전방 탐색(`(?=...)`)을 사용해 법령은 `\n(?=제\d+조)`, 판결문은 `\n(?=【)` 등을 기준으로 크게 나눕니다.
  2. **동적 프로파일링**:
     1차 분할된 블록 길이들의 80% 백분위수를 측정하여 동적 청크 기준 사이즈를 계산합니다. (안전을 위해 최소 500자 ~ 최대 5,000자 한도로 필터링)
  3. **2차 분할 (임베딩 유사도 청킹)**:
     1차 블록 중 **1,500자를 초과하는 거대 단락들만 선별**하여 내부 문장들 간의 **코사인 유사도**를 비교합니다. 유사도가 설정값(0.80) 이하로 떨어지는 주제 전환 지점(Topic Shift)에서 세부 청크로 분할합니다.

### [Step 3] 텍스트 임베딩 (Embedding)
* **역할**: 문자를 1,024차원의 실수 배열(벡터)로 변환합니다. (`bge-m3` 가속 사용)

### [Step 4] 벡터 DB 저장 (Vector Storage v2)
* **역할**: 변환된 벡터와 메타데이터를 로컬 **`data/chroma_db_v2`** 경로에 물리적으로 격리하여 저장합니다.

## 💬 2단계: 질의응답 및 추론 (Inference Lifecycle v2)
사용자가 웹 화면(`app_v2.py`)에서 질문을 입력했을 때 작동하는 삼중 가드레일 시스템의 흐름입니다.

* **지식 검색**: `Chroma DB v2` 저장소에서 질문 벡터와 가장 거리가 가까운 **상위 10개(Top 10)의 청크**를 검색합니다.
* **3차 가드레일 검사**: 거리 점수(Distance Score)가 임계값(**1.1**)을 초과하는 청크는 관련성이 없다고 판단하여 제외(Cutoff)합니다.
* **최종 추론**: 가드레일을 통과한 청크들에 한해 인라인 출처 표기(`[출처 1]`)가 첨부된 환각 없는 최종 답변을 생성합니다.
