# 로컬 Legal RAG 시스템 파이프라인 워크플로우 가이드

본 문서는 AI Hub 법률 데이터셋을 가공하여 로컬 벡터 데이터베이스를 구축하고, 삼중 가드레일을 통해 질문에 답변하는 RAG(Retrieval-Augmented Generation) 시스템의 전체 수명 주기(Lifecycle)와 단계별 순서를 설명합니다.

---

## 📌 RAG 전체 아키텍처 흐름도

```mermaid
flowchart TD
    %% Phase 1: Ingestion
    subgraph Ingestion_Phase [1단계: 지식 베이스 구축 (ingest.py)]
        A[원천 데이터 CSV + JSON 메타데이터] --> B[1. 파싱 Parser: 본문 결합 및 메타데이터 매핑]
        B --> C[2. 청킹 Splitter: 800자 단위 분할, 100자 오버랩]
        C --> D[3. 임베딩 Embedding: Ollama bge-m3 호출]
        D --> E[4. 벡터 적재 Vector Store: Chroma DB에 2,000개 배치 단위 저장]
    end

    %% Phase 2: Inference
    subgraph Inference_Phase [2단계: 질의응답 및 추론 (app.py)]
        F[사용자 질문 입력] --> G{1차 가드레일: 법률 의도 판단?}
        G -- No --> H[즉시 차단 및 거절 메세지 반환]
        G -- Yes --> I[bge-m3로 질문 임베딩 변환]
        I --> J[Chroma DB 유사도 검색 Top 3]
        J --> K{3차 가드레일: 검색 스코어 <= 1.1?}
        K -- No --> L[참고 문서 배제 / 빈 근거 전달]
        K -- Yes --> M[가드레일 통과한 근거 문서 확보]
        M --> N[2차 가드레일 & LLM: gemma4:e2b 호출]
        L --> N
        N --> O[출처 인라인 표기 및 모법 답변 생성]
    end

    E -.-> J
```

## 🗂️ 1단계: 지식 베이스 구축 (Ingestion Lifecycle)
`python ingest.py`를 실행할 때 백엔드에서 수행되는 데이터 변환 순서입니다.

### [Step 1] 파싱 및 메타데이터 결합 (Parsing)
* **역할**: 흩어져 있는 원천 법률 텍스트와 사건 정보를 하나로 묶어 정형화합니다.
* **상세**: 
  1. `01.원천데이터` 폴더의 CSV 파일들을 한 문장씩 읽어 전체 판례 본문(`full_text`)으로 병합합니다.
  2. 사전에 색인해 둔 `02.라벨링데이터` 폴더의 JSON 파일들과 1:1 매칭하여 **사건번호, 판례명, 선고일자, 문서 유형**을 가져옵니다.
  3. 만약 텍스트 내용이 없는 빈 파일이라면 메모리 낭비를 막기 위해 제외합니다.

### [Step 2] 청킹 / 텍스트 분할 (Chunking)
* **역할**: 책 한 권 수준의 법률 문서를 LLM이 소화할 수 있는 최적의 크기로 쪼갭니다.
* **상세**:
  1. `RecursiveCharacterTextSplitter`를 통해 타겟 크기 **800자**, 겹침 구간(Overlap) **100자**로 분할합니다.
  2. 법률 데이터의 특성을 살리기 위해 **`\n제` (조항 표시)**, **`\n\n` (문단 분리)**를 최우선 순위 구분 기호로 삼아 최대한 의미 구조를 보존하며 쪼갭니다.
  3. 쪼개진 개별 조각(Chunk)마다 원본 문서의 사건 정보(메타데이터)를 복사하여 꼬리표로 붙입니다.

### [Step 3] 텍스트 임베딩 (Embedding)
* **역할**: 문자(텍스트)를 컴퓨터가 수학적으로 비교할 수 있는 좌표값(벡터)으로 변환합니다.
* **상세**:
  1. Ollama에서 실행 중인 다국어 최상위 모델인 `bge-m3` 임베딩 API를 호출합니다.
  2. 각 청크의 텍스트가 1,024차원의 실수 배열(Float vector)로 인코딩됩니다. (이 단계는 100% GPU 가속을 활용해 처리됩니다.)

### [Step 4] 벡터 DB 저장 (Vector Storage)
* **역할**: 변환된 벡터와 원본 텍스트, 메타데이터를 로컬 디스크에 색인 데이터베이스로 저장합니다.
* **상세**:
  1. 데이터가 너무 많아 한꺼번에 전송 시 오류가 발생하는 것을 막기 위해 **2,000개 단위의 배치(Batch)**로 쪼개어 Chroma DB에 씁니다.
  2. 적재가 성공적으로 완료되면 프로젝트 디렉토리 아래 `data/chroma_db` 경로에 바이너리 형태의 DB 파일들이 안전하게 보존됩니다.

## 💬 2단계: 질의응답 및 추론 (Inference Lifecycle)
사용자가 웹 화면에서 질문을 입력했을 때, 삼중 가드레일 시스템이 작동하는 흐름입니다.

### [Step 1] 1차 가드레일: 질문 의도 분류 (Intent Filter)
* **역할**: 법률과 무관한 일반 질문(날씨, 요리법, 사적인 질문 등)을 RAG 파이프라인 작동 전에 즉시 필터링합니다.
* **상세**: 
  * 메인 LLM(`gemma4:e2b`)을 통해 질문의 카테고리가 대한민국 사법, 헌법, 법령 등과 관련이 있는지 분류합니다.
  * 관련이 없다고 판단(`NO`)되면 DB 조회 및 메인 연동을 건너뛰고 **"본 서비스는 법률 질의응답 서비스입니다..."**라는 안내와 함께 즉시 답변을 반환하여 리소스를 아낍니다.

### [Step 2] 지식 검색 및 3차 가드레일: 유사도 컷오프 (Similarity Search & Cutoff)
* **역할**: 질문에 맞는 정확한 판례/법령 문서를 찾아내고, 질문과 전혀 무관한 엉뚱한 정보들을 원천 배제합니다.
* **상세**:
  1. 사용자 질문을 `bge-m3` 임베딩 모델을 통해 벡터로 변환합니다.
  2. Chroma DB에서 질문 벡터와 가장 거리가 가까운 **상위 3개(Top 3)의 청크**를 검색합니다.
  3. **3차 가드레일 검사**: Chroma DB가 산출한 거리 점수(Distance Score)를 검사하여 설정해 둔 임계값(**1.1**)을 초과하는 청크는 "유사성이 낮다"고 판단하여 제외(Cutoff)합니다. 


### [Step 3] 2차 가드레일: 시스템 프롬프트 환각 제어 (Hallucination Defender)
* **역할**: 남은 법률 근거 문서만을 토대로 신뢰할 수 있는 법적 조언을 생성하며, 정보가 없으면 답변을 거부하게 만듭니다.
* **상세**:
  1. 가드레일을 통과해 선별된 법률 문서들(Context)을 조합하여 LLM에게 전달합니다. (만약 3차 가드레일에 의해 통과된 문서가 0개라면 즉시 답변 생성을 거부합니다.)
  2. **시스템 프롬프트 지침**: "반드시 제공된 참고 문서의 내용에만 철저히 기반하여 답변하라. 문서 안에서 확실한 근거를 찾을 수 없다면 조항을 유추하거나 지어내지 말고 오직 표준 거절 문장(`제공된 법률 데이터 내에서는 관련 내용을 찾을 수 없어 답변이 불가능합니다.`)만 대답하라"는 초강력 패널티 지침을 주입합니다.

### [Step 4] 인라인 출처 표기 및 답변 생성 (Citation Generation)
* **역할**: 신뢰성을 보장하기 위한 출처 표기가 첨부된 최종 답변을 화면에 렌더링합니다.
* **상세**:
  1. LLM은 답변의 근거가 되는 구절 끝에 **`[출처 1]`, `[출처 2]`** 형식의 인라인 숫자를 자동 표기합니다.
  2. 백엔드 서버는 답변 텍스트와 함께 사용된 청크의 메타데이터(사건번호 및 상세 내용)를 매핑하여 클라이언트로 반환하고, 웹 UI는 이를 깔끔한 **아코디언 카드** 형태로 사용자에게 노출합니다.

## 🛠️ 추후 관리 및 확장 요령

* **데이터 업데이트**: 
  새로운 판례 및 법률 자료가 추가되었을 경우, 기존 원천 데이터 디렉토리에 파일을 집어넣고 `python ingest.py`를 다시 실행해 주는 것만으로 최신 임베딩 정보가 반영된 정교한 법률 AI 비서 엔진을 유지할 수 있습니다.
